Aegis Documentation

Built-in guide for configuring and operating the Aegis Modbus replication system.

Overview

Aegis is a single-binary Modbus memory backplane with an integrated replication engine. It shields downstream Modbus clients from upstream device instability by holding the last known good register values in an in-process memory store.

Aegis polls one or more upstream Modbus TCP devices and republishes the data through configurable Modbus TCP servers. Downstream clients always read from the in-process store — not directly from the upstream devices.

Authentication

All WebUI pages and API endpoints require a valid login session. Authentication is always enforced.

Default Credentials

When no auth section is present in config.yaml, the default credentials are:

Usernameadmin
Passwordadmin

On first login with the default password, you will be redirected to the Change Password page. You must set a new password before accessing the dashboard.

Custom Credentials

To set custom credentials, add an auth section to config.yaml: 

auth:
  username: admin
  password_hash: "$2a$10$..."

password_hash must be a bcrypt hash. Generate one with: htpasswd -nbB admin mypassword

Sessions

Sessions are stored in memory only and expire after 8 hours. All active sessions are lost when the process restarts.

Runtime Controls

The System Controls bar at the top of the dashboard provides the following actions:

ControlDescription
Apply Config Saves the current working configuration to disk and atomically restarts the runtime with the new config. The WebUI server itself is never restarted.
Restart Runtime Reloads config.yaml from disk and rebuilds the runtime. Useful after manually editing the config file.
Export Config Downloads the currently active config.yaml to your browser.
Import Config Uploads a .yaml or .yml file, validates it, writes it to disk, and restarts the runtime.

Runtime States

The status indicator in the header reflects the current engine state:

StateMeaning
StoppedNo replicator units are running. No Modbus TCP listener is active.
StartingThe runtime is being built and port binding is in progress.
RunningAll pollers and Modbus TCP adapters are active.
StoppingThe runtime is shutting down gracefully.
ErrorThe runtime encountered an error (e.g. config validation failure, port binding failure). The error message is shown below the status indicator.

Device Setup

Each device in Aegis is made up of three parts:

  • Source — the Modbus device being polled.
  • Reads — the registers or coils to read from the source.
  • Target — the Modbus server that republishes the data.

Source Configuration

Configure the upstream Modbus device to poll.

Field Description
Device Name User-friendly name used to identify the device.
Endpoint TCP address of the Modbus device.
Format: ipaddress:port
Example: 192.168.1.100:502
Unit ID Modbus slave ID of the source device.
Timeout Maximum time to wait for a device response.

Reads Configuration

Configure the polling schedule and register ranges.

Field Description
Function Code Supported function codes:
  • FC1 – Read Coils
  • FC2 – Read Discrete Inputs
  • FC3 – Read Holding Registers
  • FC4 – Read Input Registers
Address Starting Modbus address.
Quantity Number of registers or coils to read.
Interval Polling interval in milliseconds.

Target Configuration

Configure the Modbus replication server that exposes polled data.

Field Description
Port TCP port where the replicated Modbus server will run. Default: 502
Unit ID Modbus unit ID exposed by Aegis for data registers. Range: 1–255.
Status Unit ID Unit ID used for the status memory block. Must differ from the data Unit ID on the same port. Defaults to 100 when not specified.
Status Slot Slot index representing the device inside the status block.
Each device must have a unique Status Slot within the same Status Unit ID.
Mode Authority mode controlling how Aegis handles client reads and writes:
  • A (Standalone) — Client writes allowed; reads always served regardless of upstream health.
  • B (Strict) — Default. Client writes rejected. Reads are gated by upstream health: returns 0x0B on timeout, forwards upstream exception codes.
  • C (Buffered) — Client writes rejected; reads always served even when upstream is unhealthy. Health is still tracked and reported.

Authority Modes

The Target Mode field controls how Aegis handles Modbus client read and write requests. Authority is defined per device (per replicator unit), not globally.

Mode Writes Reads
A — Standalone Allowed Always served; upstream health is not checked.
B — Strict (default) Rejected (0x01) Gated by upstream health: 0x0B on timeout; upstream exception code forwarded on Modbus error.
C — Buffered Rejected (0x01) Always served (last known good data); upstream health is still tracked and reported.

Read Coverage

Regardless of mode, reads that fall outside the bounding range of all configured read blocks return 0x02 (Illegal Data Address). In Mode B, reads that fall within the bounding range but are not covered by any read block also return 0x02; in Modes A and C they return zero-filled values.

Device Status

The Device Status panel shows real-time health metrics read from the in-process status memory block.

Field Description
Health Current device health state:
  • UNKNOWN — not yet polled since startup.
  • OK — last poll succeeded.
  • ERROR — last poll failed (timeout, exception, or disconnection).
Online True when health is OK (last poll succeeded).
Requests Total Lifetime count of poll requests sent to the upstream device.
Responses Valid Lifetime count of successful poll responses received.
Timeouts Lifetime count of poll timeouts.
Transport Errors Lifetime count of transport-level errors (connection resets, broken pipes, EOF, etc.).
Seconds in Error Seconds continuously in a non-OK health state. Reset to 0 on recovery.
Consecutive Fail (current) Current streak of consecutive failed polls.
Consecutive Fail (max) Lifetime maximum consecutive failure streak.

Status Registers (Modbus)

Each device in Aegis is assigned a Status Slot. Every slot reserves 30 Modbus registers that contain the device's health and runtime statistics.

External Modbus clients can read these registers using the configured:

  • Status Unit ID
  • Status Slot

Status Unit ID

The Status Unit ID defines the Modbus slave ID that exposes the status memory block.

Example:
Status Unit ID = 100
Clients must query Unit ID 100 to read device status.

Status Slot

Each device occupies one Status Slot. Slots are assigned sequentially starting from 0.

Device Slot
Device A0
Device B1
Device C2

Each slot contains 30 registers.

Register Addressing

The register address for a slot is calculated as:

slot_start = slot_number × 30

Slot Register Range
00 – 29
130 – 59
260 – 89
390 – 119

Register Layout

All offsets are relative to slot_start = slot_number × 30.

OffsetNameDescription
0Header Word 0Magic bytes — always 0x4147
1Header Word 1Magic + block index — always 0x53XX where the low byte contains the slot index (0–255)
2Health Code0=Unknown, 1=OK, 2=Error
3Last Error CodeModbus exception code or transport error; 0 when healthy
4Seconds In ErrorSeconds continuously in a non-OK state; reset to 0 on recovery
5–12Device Name (8 registers)16-character ASCII device name, 2 characters per register (high byte first), character pairs 1–2 through 15–16
13–19ReservedAlways zero. Reserved for future protocol use.
20–21Requests Total32-bit lifetime poll request count (low word at 20, high word at 21)
22–23Responses Valid32-bit lifetime successful poll count
24–25Timeouts32-bit lifetime timeout count
26–27Transport Errors32-bit lifetime transport error count
28Consecutive Fail (curr)Current consecutive failure streak
29Consecutive Fail (max)Lifetime maximum consecutive failure streak

Example Modbus Read

To read the status of Slot 2:

Unit ID100
Function CodeFC3
Address60
Quantity30

This will return all 30 status registers for that device.

Purpose

These registers allow SCADA or monitoring systems to detect:

  • Device offline state
  • Communication errors
  • Timeouts
  • Runtime health

This enables external systems to monitor device health without needing the WebUI.

Troubleshooting

Device not responding
Check the endpoint IP and port. The endpoint must be a valid IPv4 address (e.g. 192.168.1.100:502); hostnames are not supported.
Invalid responses
Verify the correct source unit ID and register addresses.
Timeout errors
Increase the source timeout or verify network connectivity.
Clients receive 0x0B (Gateway Target Device Failed to Respond)
The device is in Mode B (Strict) and the upstream device is not responding. Check upstream device health or switch to Mode C (Buffered) if stale data is acceptable.
Clients receive 0x01 (Illegal Function) on writes
Writes are rejected in Mode B and Mode C. Switch to Mode A (Standalone) to allow client writes.
Runtime fails to start (port binding error)
Another process is already using the configured Modbus TCP port. Free the port or configure a different target port.
Forgot password / locked out
Remove the auth section from config.yaml and restart Aegis. The default credentials (admin / admin) will be restored and you will be prompted to set a new password.
